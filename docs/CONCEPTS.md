in feaweld beyond the classical IIW / ASME / DNV / AWS workflow. It collects
mermaid architecture diagrams, high-resolution concept images (SVG + PNG),

## Architecture & pipeline

The top-level `run_analysis` orchestrator (`feaweld.pipeline.workflow`) composes
every module around a single `AnalysisCase` pydantic model. It walks through
geometry → defects → mesh → solver → postprocess → multiaxial → fatigue UQ →
fracture mechanics → report. Every new feature plugs in behind a config flag
so existing YAML cases continue to parse unchanged.

```mermaid
flowchart TD
    YAML[AnalysisCase YAML] --> LOAD[load_case]
    LOAD --> MAT[core.materials.Material]
    MAT --> GEO[geometry.joints / joints3d / spline_joints / fastener_welds]
    GEO --> DEFECT{DefectConfig.enabled?}
    DEFECT -- yes --> POP[defects.population.sample_iso5817_population]
    POP --> INSERT[defects.insertion.insert_all]
    DEFECT -- no --> MESH
    INSERT --> MESH[mesh.generator.generate_mesh]

    MESH --> MPCHECK{multipass + thermal?}
    MPCHECK -- yes --> MPSOLVE["solve_multipass_thermal\n(birth-death, interpass cooldown)"]
    MPSOLVE --> FEA[FEAResults]
    MPCHECK -- no --> BACKEND{SolverConfig.backend}

    BACKEND -->|fenics| FE[FEniCSBackend]
    BACKEND -->|calculix| CC[CalculiXBackend]
    BACKEND -->|jax| JB[JAXBackend]
    BACKEND -->|neural| NB[NeuralBackend]
    FE --> FEA
    CC --> FEA
    JB --> FEA
    NB --> FEA

    FEA --> POST[postprocess.hotspot / dong / linearization / ...]
    POST --> MA{multiaxial_criterion?}
    MA -- yes --> MAX[postprocess.multiaxial]
    MAX --> FATIGUE
    MA -- no --> FATIGUE[fatigue.rainflow + miner + sn_curves]
    FATIGUE --> UQ[_apply_fatigue_uq: lognormal bands]
    UQ --> K{compute_k_factors?}
    K -- yes --> JI[fracture.j_integral_2d / 3d]
    K -- no --> PF_CHECK

    FEA --> PF_CHECK{phase_field.enabled?}
    PF_CHECK -- yes --> PF["solve_phase_field\n(TRI3/TET4, energy split)"]

    JI --> DT{digital_twin.enabled?}
    DT -- yes --> SIFTAB["SIF table export\nfor EnKF assimilation"]

    PF --> REPORT[pipeline.report.generate_report]
    SIFTAB --> REPORT
    JI --> REPORT
    PF_CHECK -- no --> REPORT
    REPORT --> OUT[WorkflowResult + HTML report]
```

Source: `src/feaweld/pipeline/workflow.py`, tests in `tests/test_workflow_integration.py`.

## Differentiable FEA

### JAX backend (`feaweld.solver.jax_backend.JAXBackend`)

A pure-JAX finite element solver supporting linear-elastic static and
incremental elasto-plastic analyses on TRI3 / TET4 meshes. Because the
entire assembly + linear solve lives in `jax.numpy`, gradients of any
scalar QoI (compliance, displacement, stress peak) with respect to
material properties, loads, or geometry flow through via `jax.grad`.
The optional `constitutive` parameter accepts a batched
`JAXConstitutiveModel` so the backend can run J2 plasticity or FCC
crystal plasticity with autodiff-generated tangents.

<img src="images/jax_backend_flow.svg" alt="JAX backend differentiable pipeline" width="100%">

## Constitutive models

The dormant `ConstitutiveModel` ABC at `src/feaweld/solver/constitutive.py`
is finally wired into a backend via a parallel batched `JAXConstitutiveModel`
protocol at `src/feaweld/solver/jax_constitutive.py`. Three concrete models

## Fracture mechanics

### Phase-field fracture (Bourdin/Francfort AT2)

A variational phase-field solver built on the JAX backend. Damage d(x)
evolves in a staggered u-d scheme, with the elastic energy degraded by
`g(d) = (1 - d)^2 + eta`. The regularization length `l0` controls the
diffuse-crack width — the solver pre-checks that the mesh satisfies
`h <= l0 / min_mesh_ratio`.

<img src="images/phase_field_schematic.svg" alt="Phase-field AT2 functional schematic" width="100%">

<img src="animations/phase_field_crack_propagation.gif" alt="Phase-field crack propagation" width="80%">

### Staggered solver algorithm

The phase-field solver uses a classic staggered alternating minimization
at each load step. The displacement sub-problem (u-step) solves with
the current damage-degraded stiffness; the damage sub-problem (d-step)
uses the updated elastic energy density as a driving force.

```mermaid
sequenceDiagram
    participant LC as Load Controller
    participant U as u-step (displacement)
    participant H as History Field
    participant D as d-step (damage)

    LC->>LC: Apply load fraction lambda = step / n_steps
    loop Staggered iterations (max_staggered_iters)
        LC->>U: Assemble K_u(d) with g(d) = (1-d)^2 + k
        U->>U: Solve K_u u = f(lambda)
        U->>H: Compute psi_el per element
        Note over U,H: Energy split selects psi_plus (Miehe/Amor)
        H->>H: H = max(H, psi) [monotone, irreversible]
        H->>D: K_d = (Gc/l0 + 2H)M + Gc*l0*L
        D->>D: Solve K_d d = f_d, clamp [0,1], d_new >= d_old
        D-->>LC: Converged if ||d_new - d_old|| < tol
    end
    LC->>LC: Record reaction force, store damage snapshot
```

### TRI3 / TET4 element dispatch

The solver supports both 2D plane-strain (TRI3) and 3D (TET4) meshes.
Element type determines the B-matrix layout, DOFs per node, constitutive
tensor dimension, and damage mass matrix pattern.

```mermaid
flowchart TD
    START["solve_phase_field(mesh, material, load_case, config)"]
    START --> CHECK{mesh.element_type}
    CHECK -->|TRI3| TRI["2D Plane Strain"]
    CHECK -->|TET4| TET["3D"]
    CHECK -->|other| ERR["NotImplementedError"]

    TRI --> T2["B: (n_elem, 3, 6)\nDOFs/node: 2\nC: 3x3 plane-strain\nMass: (A/12) scaled 3x3"]
    TET --> T3["B: (n_elem, 6, 12)\nDOFs/node: 3\nC: 6x6 full 3D\nMass: (V/20) scaled 4x4"]

    T2 --> ASM2["_assemble_Ku_tri3\n_assemble_Kd_fd_tri3"]
    T3 --> ASM3["_assemble_Ku_tet4\n_assemble_Kd_fd_tet4"]

    ASM2 --> STAG["Staggered u-d solver loop"]
    ASM3 --> STAG
```

### Energy decomposition

The `EnergyDecomposition` enum controls which elastic energy density
drives the damage evolution. The **spectral** split (Miehe et al. 2010)
decomposes the strain tensor via eigenvalues, driving damage only under
tension. The **volumetric-deviatoric** split (Amor et al. 2009) separates
bulk and deviatoric contributions. Both prevent spurious damage under
compression.

```mermaid
stateDiagram-v2
    [*] --> ComputeStrain : eps = B * u_elem

    state EnergyDecomposition {
        [*] --> SelectSplit
        SelectSplit --> NONE : energy_split = "none"
        SelectSplit --> SPECTRAL : energy_split = "spectral"
        SelectSplit --> VOL_DEV : energy_split = "volumetric_deviatoric"

        NONE --> FullEnergy : psi = 0.5 * eps^T C eps
        SPECTRAL --> Eigendecomp : eigenvalues of strain tensor
        Eigendecomp --> MieheSplit : psi+ = lam/2*max(tr,0)^2 + mu*sum(max(eps_i,0)^2)
        VOL_DEV --> AmorSplit : psi+ = kappa/2*max(tr,0)^2 + mu*(eps_dev:eps_dev)
    }

    FullEnergy --> HistoryUpdate : H = max(H, psi)
    MieheSplit --> HistoryUpdate : H = max(H, psi+)
    AmorSplit --> HistoryUpdate : H = max(H, psi+)

    HistoryUpdate --> DamageSolve : K_d(H) d = f_d(H)
    DamageSolve --> [*]
```

### J-integral (2D + 3D) and interaction integral

Domain-form J-integral evaluated with a q-function weighting (inner
plateau → linear decay → zero cutoff). An interaction integral with
Williams-series auxiliary fields separates mixed-mode K_I / K_II
contributions. The 3D version slices the domain perpendicular to a
crack-front `WeldPath` at `n_front_samples` arc-length positions.

## Multi-scale material modeling

### FFT homogenization (Moulinec-Suquet)

Iterates the Lippmann-Schwinger equation in Fourier space on a voxel
RVE to compute the effective 6×6 Voigt stiffness. Used by
`multiscale.micro.homogenized_stiffness_from_rve` to replace the
closed-form Hall-Petch branch when an RVE `.npz` file is provided via
`MaterialConfig.rve_path`.

## Machine learning surrogates

### Bayesian deep-ensemble fatigue surrogate

Five Flax MLPs with a heteroscedastic `(mean, log_var)` head are trained
with different seeds. Predictions decompose uncertainty into epistemic
(ensemble disagreement) and aleatoric (learned noise) bands. OOD samples
show visibly widened epistemic uncertainty — the hallmark of working
deep ensembles.

<img src="images/bayesian_ensemble_uq.svg" alt="Bayesian deep-ensemble uncertainty bands" width="100%">

Source: `src/feaweld/ml/bayesian_surrogate.py`.
Tests: `tests/test_bayesian_surrogate.py`.

### Active learning over parametric studies

`ActiveLearningLoop` wraps `pipeline.study.Study` and picks the next
case via `max_variance` or `expected_improvement` on top of a Bayesian
surrogate. Epistemic-only std (not total) drives the acquisition — the
theoretically correct choice because aleatoric uncertainty is
irreducible.

### Neural operator surrogate backend (`NeuralBackend`)

A fixed-mesh DeepONet loads trained parameters from a `.msgpack` file
and returns a displacement-field prediction in under 10 ms. A SHA-256
mesh hash guards against topology drift and raises with a clear
retrain message if the incoming mesh does not match the trained one.

<img src="images/deeponet_architecture.svg" alt="DeepONet architecture" width="100%">

```mermaid
flowchart LR
    A[AnalysisCase template] --> S[pipeline.study.Study<br/>random parameter sweep]
    S --> GT{ground truth backend}
    GT --> JAX[JAXBackend]
    GT --> FE[FEniCSBackend]
    JAX --> DATA[(X, y) training set<br/>load params to displacement field]
    FE --> DATA
    DATA --> TR[Flax DeepONet<br/>branch + trunk networks]
    TR --> LOSS[MSE training<br/>optax.adam]
    LOSS --> SAVED[params.msgpack + meta.json<br/>mesh_hash stamped]
    SAVED --> NB[NeuralBackend.load_model]
    NB --> INFER[solve_static in ~10 ms<br/>mesh hash guard]
```

## Digital twin & assimilation

### EnKF crack-length assimilation

`CrackEnKF` is a stochastic Ensemble Kalman Filter driving a Paris-law
forward model. Given noisy strain-gauge observations, it tracks the
crack length distribution over 1000+ cycle blocks with < 10 % RMSE on
synthetic twins.

### Multi-state EnKF (joint crack + Paris parameter estimation)

The `MultiStateCrackEnKF` extends the scalar filter to jointly estimate
crack length `a`, Paris coefficient `log(C)`, and exponent `m`. The state
vector per ensemble member is `x = [a, log(C), m]`. During prediction,
only `a` evolves via `da/dN = exp(logC) * dK(a)^m`; `logC` and `m` are
static parameters refined through observations. Adaptive inflation
prevents ensemble collapse under strongly informative measurements.

```mermaid
sequenceDiagram
    participant M as CrackMonitor
    participant MS as MultiStateCrackEnKF
    participant PM as MultiStateParisModel
    participant OPS as Observation Operators
    participant RUL as Remaining Life

    M->>MS: Initialize ensemble (N, 3): x = [a, logC, m]
    loop Each sensor reading
        M->>MS: predict(dn = cycles_per_update)
        MS->>PM: step(state, dn) for each member
        PM->>PM: a_new = a + exp(logC) * dK(a)^m * dn
        MS->>MS: Add per-component process noise, clamp

        M->>OPS: Build operators from sensor types
        M->>MS: update(observations, obs_stds, operators)
        MS->>MS: Y_pred[i,j] = operator_j(ensemble[i])
        MS->>MS: K = P_xy * inv(P_yy + R) [matrix Kalman gain]
        MS->>MS: ensemble += (perturbed_obs - Y_pred) @ K^T
        MS->>MS: Adaptive inflation if spread < threshold

        M->>RUL: remaining_life_distribution(a_critical)
        RUL->>RUL: Propagate each member to a_critical
        RUL-->>M: mean, std, median, p05, p95
    end
```

### SIF interpolation pipeline

The `SIFInterpolator` provides tabulated stress intensity factor lookup
from pre-computed FEA results (J-integral or phase-field) or analytical
handbook formulae. Three extrapolation modes handle queries outside the
tabulated range. Residual stress contributions are integrated via the
Bueckner weight function and combined with the applied SIF.

```mermaid
flowchart TD
    subgraph Sources["SIF Data Sources"]
        JI["J-integral results"]
        HB["Handbook formula\ndK = Y * S * sqrt(pi*a)"]
    end

    subgraph Table["SIFTable"]
        TAB["crack_lengths[]\nsif_values[]"]
    end

    JI -->|"from_j_integral_results()"| TAB
    HB -->|"from_handbook(S, Y)"| TAB
    TAB --> INTERP["SIFInterpolator"]

    subgraph Extrap["Extrapolation"]
        EP["paris: fit Y*S, extend\nwith sqrt(pi*a)"]
        EC["constant: clamp\nto endpoints"]
        EZ["zero: 0 outside range"]
    end
    INTERP --> EP & EC & EZ

    subgraph Residual["Residual Stress"]
        RSP["sigma_res(z/t) profile"]
        BWF["Bueckner integral\nK_res(a)"]
    end
    RSP --> BWF
    BWF --> COMB["combined_sif()\ndK_eff = max(K_app + K_res, 0)"]
    INTERP --> COMB
    COMB --> ENKF["EnKF dK_fn"]
```

### Ensemble remaining useful life prediction

The `remaining_life_distribution` method propagates every ensemble member
forward under the Paris law until the crack reaches the critical length.
This produces a full distribution of remaining fatigue life, capturing
uncertainty in both the current crack state and the material parameters.

```mermaid
stateDiagram-v2
    [*] --> Ensemble : N members with [a_i, logC_i, m_i]

    Ensemble --> Propagate : remaining_life_distribution(a_critical)

    state Propagate {
        [*] --> Step : state = model.step(state, dn)
        Step --> Check : a >= a_critical?
        Check --> Record : yes -> lives[i] = total_cycles
        Check --> Step : no -> continue
        Step --> MaxReached : max_steps exceeded
        MaxReached --> Infinite : lives[i] = inf
    }

    Propagate --> Stats : Filter finite lives
    Stats --> Output : mean, std, median, p05, p95, fraction_survived
    Output --> [*]
```

### Defect types

Seven dataclasses covering the major weld defect families:
`PoreDefect`, `ClusterPorosity`, `SlagInclusion`, `UndercutDefect`,
`LackOfFusionDefect`, `RootGapDefect`, `SurfaceCrack`. Each implements
the `Defect` protocol (`volume`, `critical_dimension`, `description`).

<img src="images/defect_type_gallery.svg" alt="Weld defect type gallery" width="100%">

Source: `src/feaweld/defects/types.py`. Tests: `tests/test_defect_types.py`.

### Defect insertion workflow

```mermaid
flowchart TD
    C[DefectConfig.enabled] --> S{explicit_defects empty?}
    S -- yes --> DRAW[sample_iso5817_population<br/>level B / C / D]
    S -- no --> LIST[Use explicit defect list]
    DRAW --> VAL[defects.population.validate_population]
    VAL --> BAD{violations?}
    BAD -- yes --> REDRAW[Redraw up to 10 attempts]
    REDRAW --> VAL
    BAD -- no --> LIST
    LIST --> JOINT[Base joint build in Gmsh OCC]
    JOINT --> INSERT[defects.insertion.insert_all]
    INSERT --> PORE[insert_pore: boolean cut sphere]
    INSERT --> SLAG[insert_slag_inclusion: dilate + fragment]
    INSERT --> UND[insert_undercut: sweep + cut]
    INSERT --> LOF[insert_lack_of_fusion: planar fragment]
    INSERT --> CRACK[insert_surface_crack: semi-ellipsoid cut]
    PORE --> TAG[Physical groups tagged]
    SLAG --> TAG
    UND --> TAG
    LOF --> TAG
    CRACK --> TAG
    TAG --> CLEAN[removeAllDuplicates + synchronize]
    CLEAN --> MESH[generate_mesh]
    MESH --> KD[fatigue.knockdown.defect_knockdown]
    KD --> OUT[extensions defects population + worst_fat]
```

## Fatigue & post-processing

### Multi-axial fatigue criteria

Six critical-plane and invariant-based criteria operating on full
stress-tensor histories: Findley, Dang Van, Sines, Crossland,
Fatemi-Socie, McDiarmid. Plane search uses a ~200-point Fibonacci
spherical grid on the upper hemisphere.

<img src="images/multiaxial_critical_plane.svg" alt="Multi-axial fatigue critical-plane search" width="80%">

Source: `src/feaweld/postprocess/multiaxial.py`.
Tests: `tests/test_multiaxial_fatigue.py`.

### Rainflow multi-axial counting

Projects stress-tensor histories onto either a critical-plane normal
(via `n · σ · n`) or the signed maximum principal, then delegates
scalar cycle counting to the existing ASTM E1049 `rainflow_count`.

<img src="images/rainflow_multiaxial_projection.svg" alt="Rainflow multiaxial projection" width="100%">

<img src="animations/rainflow_cycle_counting.gif" alt="Rainflow cycle counting sweep" width="80%">

MP4: [`rainflow_cycle_counting.mp4`](animations/rainflow_cycle_counting.mp4)

Source: `src/feaweld/fatigue/rainflow.py`. Tests: `tests/test_rainflow_multiaxial.py`.

### Probabilistic Miner damage

`life_with_scatter_stress` + `miner_damage_with_uq` propagate
lognormal S-N scatter + Gaussian stress uncertainty through Palmgren-
Miner accumulation, returning mean / std / p05 / p95 fatigue-life
bands alongside the existing deterministic scalar keys. Wired into
`run_analysis` via `_apply_fatigue_uq`.

Sources: `src/feaweld/fatigue/sn_curves.py`, `src/feaweld/fatigue/miner.py`.
Tests: `tests/test_probabilistic_miner.py`.

### Monte Carlo convergence

<img src="animations/monte_carlo_convergence.gif" alt="Monte Carlo sampling convergence" width="80%">

MP4: [`monte_carlo_convergence.mp4`](animations/monte_carlo_convergence.mp4)

Source: `src/feaweld/probabilistic/monte_carlo.py`.

### Path extraction along splines

`extract_along_path(fea_results, path, n_samples)` uses a thin-plate-
spline RBF interpolator (reused from `singularity.submodeling`) to
sample a nodal field at arc-length positions along a curved weld toe.
Falls back to nearest-neighbor on ill-conditioned RBF systems and
records the fallback in the result metadata.

Source: `src/feaweld/postprocess/path_extraction.py`.
Tests: `tests/test_path_extraction.py`.

### Curved hotspot along a path

`hotspot_stress_along_path(fea_results, path, plate_thickness)` computes
IIW Type A / Type B hot-spot stresses at arc-length positions along
a curved weld path by sampling at `0.4t` and `1.0t` (Type A) or
`4 / 8 / 12 mm` (Type B) offsets along the Frenet principal normal.
On straight paths it reproduces the legacy discrete-node API to 1e-8.

Source: `src/feaweld/postprocess/hotspot.py`.
Tests: `tests/test_hotspot_along_path.py`.

## Multi-pass welding

### Weld sequence + thermal dispatch

`WeldSequence` composes multiple `WeldPass` entries with validated
monotone start times; `MultiPassHeatSource` dispatches to per-pass
`GoldakHeatSource` instances based on the current simulation time.
The existing single-source Goldak API is unchanged.

<img src="images/multipass_bead_stacking.svg" alt="Multi-pass bead stacking" width="100%">

<img src="animations/multipass_thermal_cycle.gif" alt="Multi-pass thermal cycle" width="80%">

### Goldak moving heat source

<img src="animations/goldak_heat_source_sweep.gif" alt="Goldak double-ellipsoid heat source sweep" width="80%">

MP4: [`goldak_heat_source_sweep.mp4`](animations/goldak_heat_source_sweep.mp4)

```mermaid
sequenceDiagram
    participant G as geometry.multipass
    participant T as solver.thermal.MultiPassHeatSource
    participant M as solver.mechanical
    participant R as residual stress

    G->>T: build_multipass_joint(base, sequence, path, profiles)
    Note over G,T: root + fill + cap beads as physical groups
    T->>T: Root pass t in [0, d1)<br/>Goldak source 1
    T->>M: temperature field
    M->>R: thermal stress + plasticity
    T->>T: Interpass cooldown
    T->>T: Fill pass t in [d1+gap, ...)<br/>Goldak source 2
```

### Multi-pass thermal solver (`solve_multipass_thermal`)

The full multi-pass thermal cycle solver orchestrates per-pass transient
solves with element birth-death activation and interpass cooldown monitoring.
Temperature state is explicitly carried between passes via the
`initial_temperature` parameter added to all `SolverBackend.solve_thermal_transient()`
implementations.

```mermaid
sequenceDiagram
    participant O as solve_multipass_thermal
    participant BD as ElementBirthDeath
    participant MH as _MaskedHeatSource
    participant BE as SolverBackend
    participant IC as InterpassCheck

    O->>O: T_current = preheat_temp (all nodes)
    O->>BD: Build per-pass birth-death from pass_N physical groups
    loop Each WeldPass (root, fill, cap, ...)
        O->>MH: Wrap Goldak source + birth-death mask
        O->>BE: solve_thermal_transient(heat_source=MH, initial_temperature=T_current)
        BE-->>O: per_pass_result (temperature history)
        O->>O: T_current = last temperature snapshot
        MH->>BD: update(torch_pos) activates elements behind torch
        alt Not last pass
            O->>O: T_max = max(T_current)
            alt T_max > interpass_temp_max
                O->>BE: solve_thermal_transient(heat_source=None, initial_temperature=T_current)
                BE-->>O: Cooled temperature field
                O->>IC: Record max_temp, cooldown_time, passed
            end
        end
    end
    opt compute_residual_stress
        O->>O: compute_thermal_stress(T_current, T_ref=ambient)
    end
```

### Element birth-death lifecycle

The `ElementBirthDeath` class manages activation of weld filler elements
as the torch passes. Dead elements receive zero heat via the `_MaskedHeatSource`
wrapper. Activation is based on signed distance from the torch along the
travel direction.

```mermaid
stateDiagram-v2
    [*] --> Dead : Element in weld group, not yet deposited

    Dead --> Dead : Torch has not reached centroid
    Dead --> Activating : signed_dist <= activation_distance

    state Activating {
        [*] --> CheckPerp : Perpendicular distance check
        CheckPerp --> Activate : perp_dist <= 3 * activation_distance
    }

    Activating --> Alive : alive_mask set to True

    Alive --> Alive : Full thermal participation
    Alive --> [*] : Simulation complete
```

### Temperature carry-forward between passes

```mermaid
flowchart TD
    subgraph Pass1["Pass 1 (Root)"]
        P1S["T_init = preheat_temp"] --> P1H["Goldak source active"]
        P1H --> P1E["T_end_pass1"]
    end
    subgraph Cool1["Interpass Cooldown"]
        C1S["T_init = T_end_pass1"] --> C1M{"T_max > limit?"}
        C1M -- yes --> C1R["Cooldown transient"]
        C1M -- no --> C1P["Skip"]
        C1R --> C1E["T_cooled"]
        C1P --> C1E
    end
    subgraph Pass2["Pass 2 (Fill)"]
        P2S["T_init = T_cooled"] --> P2H["Goldak source active"]
        P2H --> P2E["T_end_pass2"]
    end
    subgraph Pass3["Pass 3 (Cap)"]
        P3S["T_init = ..."] --> P3H["Goldak source active"]
        P3H --> P3E["T_final"]
    end
    P1E --> C1S
    C1E --> P2S
    P2E --> P3S
    P3E --> RES{"Residual stress?"}
    RES -- yes --> STRESS["sigma_res = -C : eps_thermal"]
    RES -- no --> OUT["MultiPassThermalResult"]
    STRESS --> OUT
```
